## 2. Процесс разработки

Работа шла «agentic», а не «chat-based»: я не копировал куски кода вручную,
а ставил агенту задачи и ревьюил результат.

**Этапы:**

1. **Архитектурный промпт → план.** Сначала я скормил агенту кастомный промпт
   по Hexagonal Architecture (§3). На выходе получился `PLAN.md` — доменная
   модель, слои, порты/адаптеры, этапы. Это «карта», по которой потом шла
   вся генерация.

2. **Scaffold снизу вверх (по слоям).**
   - `domain/` → `application/` → `infrastructure/` → `app/` (presentation).
   - Сначала value objects и entities (чистая логика), потом use cases,
     потом адаптеры (Supabase, AI), потом UI.

3. **Mock-first.** AI-слой сразу сделан через `IAIProvider` с `MockAIAdapter`
   по умолчанию. Так приложение работало end-to-end **без единого платного
   API-ключа**, а реальные провайдеры (OpenAI/Anthropic/Gemini) добавлялись
   как параллельные адаптеры — менять их можно одной env-переменной.

4. **Итеративный дебаг Supabase.** Бо́льшая часть правок коммитов пришлась на
   RLS/auth: async cookies в Next.js 15+, корректный server-client для
   middleware, upsert пользователя перед генерацией. Каждая проблема
   формулировалась как точечный промпт (см. §4, промпты 11–15).

5. **Финализация сдачи (этап, описанный в этом отчёте).** Добавлены error
   boundaries (чтобы не было белых экранов — требование задания), написаны
   unit-тесты, поправлен README, создан этот отчёт.

**Контур контроля качества:**
- `npm run typecheck` и `npm run lint` после каждой существенной правки;
- ручная проверка каждой сгенерированной страницы в браузере;
- review архитектурных решений (правильно ли домен зависит от инфраструктуры — нет).

## 3. Кастомный промпт по Hexagonal Architecture

Чтобы агент генерировал именно чистую DDD/Hexagonal-структуру (а не типичный
«всё в одном файле» Next.js-код), использовался кастомный системный промпт.
Его суть (полная версия воплощена в `PLAN.md`):

> Спроектируй Next.js-приложение по Domain-Driven Design и Hexagonal
> Architecture (Ports & Adapters). Доменный слой (`src/domain`) должен иметь
> **ноль** зависимостей от фреймворков и БД — только чистый TypeScript:
> entities, value objects, domain events, интерфейсы репозиториев (ports),
> доменные ошибки. Use cases (`src/application`) оркестрируют сценарии и
> зависят только от интерфейсов (`IAIProvider`, `IAuthService`, репозитории),
> а не от реализаций. Конкретные технологии (Supabase, OpenAI, Stripe) —
> это **адаптеры** в `src/infrastructure` и подставляются через DI-контейнер.
> Presentation layer (Next.js App Router) — тонкий и вызывает только use cases.
> Соблюдай SOLID, особенно Dependency Inversion и Open/Closed: новая фича =
> новый адаптер, а не правка домена.

Этот промпт задавался в начале каждой сессии генерации нового слоя, чтобы
агент не «съезжал» в процедурный стиль.

## 5. Что особенно хорошо получилось благодаря AI

- **Чистая изоляция домена.** Кастомный hexagonal-промпт удержал агента от
  «утечки» Supabase/OpenAI в доменный слой. Заменить провайдера можно одной
  env-переменной — это прямое следствие правильных портов.
- **Mock-first.** Приложение работало end-to-end без платных ключей с первого
  дня — опять же заслуга архитектуры, навязанной промптом.
- **Скорость scaffold'а.** Полная DDD-структура (десятки файлов) была
  сгенерирована за сессии, а не дни.

## 6. Что было сложно / где AI ошибался

- **RLS Supabase + Next.js 15 async cookies.** Агент несколько раз выдавал
  устаревший код (`createMiddlewareClient`, синхронные `cookies()`). Пришлось
  серией точечных промптов (9–12) довести до рабочих `createServerClient` +
  `await cookies()`. Это классическая зона, где бесплатная модель галлюцинирует
  версию API.
- **Безопасность политик.** Агент готов был согласиться на небезопасный
  `WITH CHECK (true)` — потребовался явный review-промпт (13), чтобы это поймать.
  Мораль: AI-генерированные миграции/политики безопасности нужно ревьюить
  особенно придирчиво.
- **Точность артефактов сдачи.** README расходился с реальностью (порт 3000
  вместо 8364, упоминание несуществующих тестов). Это починено вручную.

## 8. Вывод

Основной код сгенерирован связкой **KiloCode (StepFun 3.7 free) + кастомный
hexagonal-промпт**, что дало правильную архитектуру «из коробки». AI-агенты
сильны в scaffold'е и повторяющихся структурах, но **требуют контроля в зонах
версионно-чувствительных API (Next.js 15/16, Supabase) и безопасности (RLS)** —
там правки приходилось итеративно уточнять и ревьюить вручную. Mock-first на
чистых портах оказался ключом к тому, чтобы сдать рабочий MVP без затрат на
платные API.
